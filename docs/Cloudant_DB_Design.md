# IBM Cloudant DB Design

## Overview
For this project, we are utilizing **IBM Cloudant**, a fully managed, distributed NoSQL JSON document database. It is highly suited for IoT applications due to its flexible schema and excellent scalability.

## Database Details
* **Database Name:** `waste_management`
* **Type:** NoSQL Document Store
* **Format:** JSON

## Document Structure
Every time the Python simulator sends data and Node-RED processes it, a new document is inserted into Cloudant.

### Example JSON Document:
```json
{
  "_id": "6d92ca8a2305a41d99671b563e001c43",
  "_rev": "1-a3f8902c6b4122d3345e5b66d7789012",
  "Bin_ID": "Bin_001",
  "Latitude": 28.621453,
  "Longitude": 77.198302,
  "Fill_Level": 85,
  "Temperature": 32.4,
  "Timestamp": "2023-11-05T14:32:01.123456",
  "Status": "Critical"
}
```

### Field Descriptions:
1. `_id`: Unique identifier generated automatically by Cloudant.
2. `_rev`: Revision ID generated by Cloudant for document version control.
3. `Bin_ID`: String. The unique identifier of the waste bin.
4. `Latitude`: Float. GPS Latitude coordinate.
5. `Longitude`: Float. GPS Longitude coordinate.
6. `Fill_Level`: Integer (0-100). Represents the percentage of waste in the bin.
7. `Temperature`: Float. The current temperature inside the bin (Celsius).
8. `Timestamp`: String (ISO 8601 format). The exact date and time the reading was taken.
9. `Status`: String. Calculated by Node-RED. Values: `Normal` (<50%), `Moderate` (50-80%), `Critical` (>80%).

## Creating Indexes for Analytics
To effectively query this data later (e.g., for generating historical charts or finding all "Critical" bins), you would create an index in Cloudant.

**Example Query to find all critical bins:**
```json
{
   "selector": {
      "Status": {
         "$eq": "Critical"
      }
   }
}
```
